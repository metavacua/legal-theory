# Consolidated References & Bibliography: Design

## 1. Motivation

The corpus has **no structured bibliography anywhere**. A repo-wide survey (method in §2) found
zero uses of DocBook `<biblioentry>`, `<biblioref>`, `<bibliography>`, or `<footnote>` across all
859 non-paper XML files. What exists instead:

- **~89 documents** carry a `<section xml:id="works-cited">` with an `<orderedlist>` of
  informally-formatted citation strings ending in a `<link xlink:href="...">` — **5,488 links
  total**, almost all pointing at secondary aggregator sites (Justia, FindLaw, CaseMine,
  Wikipedia, law-firm blogs) rather than primary sources. The same statute or case is often
  cited from 2-3 different aggregator sites as separate, undeduplicated entries.
- Some works-cited lists contain **garbled entries** with no link at all — artifacts of the
  original Markdown→pandoc→DocBook conversion (e.g. a bare `"Systemic_Misclassification"` or a
  truncated mid-sentence fragment as a list item).
- **446 files** contain orphaned numeric footnote markers (`...Court</emphasis>.4`) glued
  directly onto the preceding text with no space and no structural link to any works-cited
  entry — vestiges of Markdown footnotes that pandoc flattened during conversion.
- The one scholarly paper has a proper `bibliography.bib` (28 entries, BibTeX), but it is
  **disconnected from its own build**: neither `latex.xsl` nor the paper's `Makefile` ever emits
  `\bibliography{}`/`\bibliographystyle{}` or runs `bibtex`/`biber`, and `html5.xsl` renders
  `<citation>KEY</citation>` as the literal bracketed string `[KEY]`, not a resolved reference.

This design produces one canonical, deduplicated, browsable **academic bibliography** for the
repository — legal sources cited in Bluebook, everything else cited in Chicago Author-Date —
generated by a repeatable script rather than assembled by hand, since the corpus is still
growing and a hand-maintained list would drift immediately.

**Sequencing, not exclusion:** resolving the 446 orphaned footnote markers back to specific
bibliography entries, and reconnecting `bibliography.bib` to the paper's own LaTeX build, are
real follow-on work planned to happen **after** this project, not features cut from it. They are
listed as next-up work in §8 rather than folded into this pass because they carry a different
risk profile — a footnote-to-citation rewrite touches the body text of 446 *existing* documents
and needs its own per-file verification pass before any mechanical rewrite, which is a distinct
project deserving its own design and plan.

## 2. Survey method (for anyone re-deriving these numbers)

Counts above came from `xmllint --xpath`/`grep -rl` sweeps over `docs/**/*.xml` (excluding
`docs/papers/**` and `docs/scripts/**`) for: `<biblioentry>`, `<biblioref>`, `<citetitle>`,
`<footnote>`, `<bibliography>`, `<citation>`, `xml:id="works-cited"`, `<link xlink:href`, and the
trailing-digit footnote-marker pattern. `docs/schema/docbook-corpus.rnc` and both `docs/xsl/*.xsl`
files were read in full to confirm neither constrains nor renders bibliography markup today.
`docs/common/shared-metadata.xml` was read as the repo's only precedent for a canonical
XInclude-able shared fragment (114+ includers, one root-typed file, relative-path references).

## 3. Citation policy (explicit, stated in the document itself)

> Primary legal sources (cases, statutes, regulations) are cited in **Bluebook (21st ed.)**
> format. All other sources — academic, technical, and secondary — are cited in **Chicago
> Author-Date (17th ed.)** format. A field not verifiable from the source corpus is marked
> `[unknown]` rather than inferred or guessed. This repository publishes analysis used in real
> legal controversies; a fabricated citation component is worse than a visibly incomplete one.

This no-fabrication rule is the load-bearing constraint of the whole design: **elevate format,
never invent facts**. Where a case name is known but no reporter citation exists anywhere in the
source text, the output is a flagged partial citation (`Case Name, [reporter citation
unknown]`), not a guessed one. Where a secondary source has no identifiable author, the output
literally says `[author unknown]`.

## 4. Output structure & location

A new DocBook 5.2 article, following the exact convention every other corpus document already
uses (shell `.xml` + `.meta.xml` companion `xi:include`-ing `docs/common/shared-metadata.xml`):

```
docs/bibliography/references.xml        # shell article
docs/bibliography/references.meta.xml   # <info> + shared-metadata include
docs/bibliography/references.html       # built output (committed, like every other doc)
```

Zero XSL or schema changes needed: the body uses only `<section>`, `<orderedlist>`,
`<listitem>`, `<para>`, and `<link xlink:href>` — the exact markup pattern every existing
works-cited section already uses and `html5.xsl` already renders correctly. `build-corpus.yml`'s
existing `find` picks up any article-rooted XML under `docs/` automatically.

Four sections, in order:

1. **Front matter** — scope/methodology (condensed §1-2) plus the citation-policy statement
   from §3, verbatim, so a reader knows exactly what convention they're looking at before
   reading a single entry.
2. **Legal Citations** — Bluebook. Cases and statutes/regulations extracted from works-cited
   entries (§5) plus the 10 case/statute/regulation entries from `bibliography.bib`. Alphabetized
   by case name / code name.
3. **Academic & Secondary Sources** — Chicago Author-Date. Generic works-cited link entries plus
   the remaining 18 `bibliography.bib` entries. Alphabetized by author surname, or by title
   where `[author unknown]`.
4. **Appendix: Entries Needing Manual Review** — works-cited listitems with no `<link>` and no
   recognizable citation pattern (the garbled conversion artifacts). Logged verbatim with their
   source file, not silently dropped, so a human can later fix, properly cite, or delete at the
   source.

**Every entry lists which document(s) cite it**, linked to that document's built `.html` page
(the same "link to built output, not source" convention `docs/index.md` already uses) — this is
the traceability a flat reference list would otherwise lose. Backlinks are resolved by walking
each shell article's `xi:include` graph, so a works-cited section living in a fragment file
correctly backlinks to its *owning shell article's* `.html`, not the fragment itself (fragments
have no independent build output).

## 5. Extraction, classification, and elevation

**Extract:** walk `docs/` (excluding `docs/papers/**`, `docs/scripts/**`) for `works-cited`
sections; separately parse `bibliography.bib` in full. For each works-cited `<listitem>`, capture
its normalized text and optional `<link xlink:href>`.

**Classify** each raw entry into exactly one bucket:

| Pattern | Bucket | Notes |
|---|---|---|
| Code name + `§` (e.g. "California Civil Code § 1550") | Statute → Legal Citations | Code name normalized to standard Bluebook abbreviation via a lookup table built from the survey's abbreviation-diversity findings (Civil Code → `Cal. Civ. Code`, etc.) |
| Strict `X v. Y` (title-case parties, literal `" v. "`, tuned against false positives like an essay titled "privacy v. transparency") | Case → Legal Citations | If a full reporter cite (e.g. `18 Cal. 3d 660`) is present in the *same* entry text, full Bluebook cite; otherwise a flagged partial cite `Case Name, [reporter citation unknown]`, source URL retained as an access note |
| `bibliography.bib` entry, case/statute/regulation type (10 entries: `gema_openai2025`, `garcia_characterai2025`, `bartz_anthropic2025`, `kadrey_meta2025`, `litchfield1984`, `copyright1976fixation`, `gdpr2016`, `ccpa2018`, `eu_database_directive1996`, `eu_pld2024`) | Legal Citations | Uses the bib's own fields, already structured |
| `bibliography.bib` entry, everything else (18 entries: arXiv papers, tools, EU reports, self-citations) | Academic & Secondary | Chicago Author-Date from existing author/year/venue fields. The 3 self-citations to in-repo documents (`mclean2025categorical`, `mclean2025encoding`, `mclean2025agpl`) resolve their URL to this repo's own built HTML page for that document, not the raw GitHub blob path currently in the `.bib` file |
| Has a `<link>`, matches no statute/case pattern | Academic & Secondary | Chicago web-citation form: `[Author/Organization]. "[Title]." [Site]. Accessed [date]. [URL].` Author = publisher/site name if identifiable, else `[author unknown]`. Date = the "accessed \<date\>" text already in the source, explicitly labeled as an access date (not a publication date, which is `[year unknown]` unless separately evidenced) |
| No `<link>`, no pattern match | Appendix | Not force-classified |

**Elevate:** render per the table above, applying `[unknown]` markers per §3's no-fabrication
rule.

**Dedup:** normalized URL (lowercase scheme+host, trailing slash stripped, `www.` prefix left
as-is) where a URL exists; normalized citation string for URL-less `bibliography.bib` entries.
Files are walked in sorted path order (deterministic, not filesystem-iteration-order-dependent),
so "first-seen" is reproducible run to run. Merging keeps the first-seen title/entry and unions
the "cited in" backlink list. The "accessed
\<date\>" suffix is stripped from display titles once deduplicated (a single access date is
meaningless once several citing documents are merged) but the raw original per-citation text is
preserved in a sidecar QA log for traceability.

## 6. Verification (this is not skippable — 5,488 entries can't be hand-checked one by one)

1. **TDD for the script**, unit-tested per stage (xi:include resolution, statute regex, case
   regex, dedup-key normalization, Bluebook/Chicago formatting) against fixtures that
   deliberately include adversarial cases *before* the happy path is considered done: a
   false-positive `" v. "` in a non-case title, a garbled no-link entry, duplicate URLs differing
   only by trailing slash or case, a statute cited with a nonstandard abbreviation, a
   `bibliography.bib` self-citation.
2. **Automated self-check invariants** the script asserts on its own output before it's allowed
   to finish: no entry lost (`len(input entries) == len(output entries) + len(appendix entries)`
   post-dedup accounting), no Legal Citations entry lacking `§` or `" v. "`, no duplicate
   normalized URLs surviving dedup, every backlink's target `.html` path actually exists on
   disk, and the emitted XML validates (`xmllint --xinclude`, `jing -c
   docs/schema/docbook-corpus.rnc`) and builds (`xsltproc --xinclude docs/xsl/html5.xsl`)
   cleanly.
3. **"Too right" tripwire.** An empty or near-empty Appendix is a red flag, not a clean result —
   the survey already found real garbled entries, so their absence from the output means the
   classifier is silently swallowing them, not that they don't exist. Any suspiciously clean
   invariant result triggers a systematic-debugging pass before the run is trusted, exactly as
   much as an outright failure would.
4. **Code review** of the script (via the repo's code-review process) before it is run
   corpus-wide.
5. **Simplify pass** on the script after it passes its tests, before the corpus-wide run.
6. **Stratified human-reviewable sample**, weighted toward the weakest-heuristic buckets
   (flagged partial case cites, generic secondary sources with `[author unknown]`), as the actual
   completion gate — passing tests is necessary, not sufficient, for content whose correctness
   can't be exhaustively assert-checked by code alone.
7. **Regeneration stays a deliberately-invoked maintenance script**, like
   `atomize_existing_document.py`, not wired into CI's auto-commit-on-push behavior — heuristic
   classification output needs a review step before it lands in a corpus used for real legal
   controversies.

## 7. Testing conventions

Follows the existing `docs/scripts/` convention: `docs/scripts/build_bibliography.py` +
`docs/scripts/tests/test_build_bibliography.py`, fixtures under
`docs/scripts/tests/fixtures/bibliography/` (mirroring the existing `docs/scripts/tests/fixtures/`
layout used by `test_convert_to_docbook.py`).

## 8. Explicitly sequenced follow-on work (next, not cut)

1. **Footnote-to-citation linking.** Verify, file by file, whether the 446 files' trailing
   footnote digits correspond positionally to works-cited ordinal entries, and if so,
   mechanically convert them into real `<footnote>`/`xref` links into this bibliography. Its own
   design pass, after this one lands.
2. **Reconnect `bibliography.bib` to the paper's LaTeX build** — add
   `\bibliography{bibliography}`/`\bibliographystyle{...}` to `latex.xsl`'s preamble and a
   `bibtex`/`biber` step to the paper's `Makefile`; separately decide whether `html5.xsl`'s
   `<citation>` template should resolve against this same consolidated bibliography instead of
   emitting a bare bracketed key.
3. **Clean up the Appendix entries** at the source — either supply a real citation/link or
   delete the garbled listitem from its origin document.
4. **Normalize the citation-abbreviation diversity** documented in the survey (e.g. `Corp. Code §`
   vs `Corporations Code §` vs `CORP §` for the same code) as a corpus-wide style pass, independent
   of the bibliography itself.
